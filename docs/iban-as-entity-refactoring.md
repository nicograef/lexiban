# Refactoring-Plan: IBAN als Entität statt Validierungsergebnis

> **Ziel**: Die `ibans`-Tabelle soll eindeutige IBANs speichern (eine Zeile pro IBAN), nicht mehrfache Validierungsergebnisse. Wiederholte Anfragen für dieselbe IBAN sollen das gespeicherte Ergebnis als Lookup zurückgeben, statt erneut zu validieren.

---

## 1. Ist-Zustand — Analyse

### Aktuelles Schema

```sql
CREATE TABLE ibans (
    id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    iban       VARCHAR(34) NOT NULL,
    bank_name  VARCHAR(255),
    bank_identifier VARCHAR(20),
    valid      BOOLEAN NOT NULL,
    validation_method VARCHAR(20) NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_ibans_iban ON ibans (iban);
```

### Was passiert heute bei einer Anfrage?

```
POST /api/ibans { "iban": "DE89370400440532013000" }

1. Controller empfängt Request
2. IbanValidationService.validate() → Mod-97 + BLZ-Lookup
3. Falls bankName == null → ExternalIbanApiService.validate() (openiban.com)
4. ibanRepository.save(new Iban(...))  ← IMMER ein neuer INSERT
5. Response zurück
```

### Problem

- **Duplikate**: Sendet ein Nutzer die gleiche IBAN 10-mal, entstehen 10 Zeilen in der DB.
- **Unnötige Arbeit**: Jede Anfrage durchläuft die volle Validierung + ggf. externen API-Call, obwohl das Ergebnis deterministisch ist (die gleiche IBAN ist immer gültig oder ungültig).
- **Semantik-Mismatch**: Die Tabelle heißt `ibans`, enthält aber keine IBANs, sondern Validierungs-Logs. Der Index `idx_ibans_iban` existiert, wird aber nie für Lookups genutzt.
- **Kein natürlicher Schlüssel**: Der `BIGINT id` ist ein synthetischer Surrogate Key ohne fachliche Bedeutung. Die IBAN selbst ist der natürliche, fachlich eindeutige Identifikator.

---

## 2. Soll-Zustand — Konzept

### Neues Datenmodell

Die IBAN selbst wird zum Primary Key. Jede IBAN existiert **genau einmal**. Die Tabelle wird zum **Lookup-Cache** für Validierungsergebnisse.

```sql
CREATE TABLE ibans (
    iban              VARCHAR(34) PRIMARY KEY,      -- natürlicher PK
    bank_name         VARCHAR(255),
    bank_identifier   VARCHAR(20),
    valid             BOOLEAN NOT NULL,
    reason            VARCHAR(255),                 -- Grund bei ungültiger IBAN
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**Änderungen gegenüber heute:**

| Aspekt                 | Vorher                                 | Nachher                                 |
| ---------------------- | -------------------------------------- | --------------------------------------- |
| Primary Key            | `id BIGINT` (Surrogate)                | `iban VARCHAR(34)` (Natural Key)        |
| Duplikate              | Erlaubt (jede Anfrage = neuer Eintrag) | Unmöglich (PK-Constraint)               |
| Index `idx_ibans_iban` | Separater Index                        | Entfällt — PK ist automatisch indiziert |
| `validation_method`    | Vorhanden                              | Entfällt komplett                       |
| `reason`               | Nicht vorhanden                        | Neues Feld (nullable) für Fehlergrund   |
| `updated_at`           | Nicht vorhanden                        | Neues Feld für Cache-Frische            |
| `id`-Spalte            | Vorhanden                              | Entfällt komplett                       |

### Neuer Ablauf bei einer Anfrage

```
POST /api/ibans { "iban": "DE89370400440532013000" }

1. Controller empfängt Request
2. Normalisierung: Leerzeichen entfernen, Uppercase
3. ── NEU: Lookup ──
   ibanRepository.findById(normalizedIban)
   → Falls vorhanden: Gespeichertes Ergebnis direkt zurückgeben (kein Validate!)
4. ── Nur bei Cache-Miss: ──
   IbanValidationService.validate() → Mod-97 + BLZ-Lookup
   Falls bankName == null → ExternalIbanApiService.validate()
   ibanRepository.save(new Iban(...))
5. Response zurück
```

**Vorteile:**

- Wiederholte Anfragen sind ein simpler PK-Lookup (O(log n), quasi instant)
- Kein überflüssiger externer API-Call für bereits bekannte IBANs
- Saubere Semantik: die Tabelle enthält tatsächlich IBANs
- Weniger Datenbank-Schreiboperationen

## 3. Datenbankwahl — Ist PostgreSQL die richtige Wahl?

### Analyse des Zugriffmusters (nach dem Umbau)

| Eigenschaft      | Wert                                                                     |
| ---------------- | ------------------------------------------------------------------------ |
| Primärer Zugriff | Key-Value-Lookup (IBAN → Validierungsergebnis)                           |
| Schlüssel        | String, 15–34 Zeichen, eindeutig                                         |
| Wert             | Struct mit 5–6 Feldern                                                   |
| Schreibmuster    | Write-once (erste Validierung), danach nur Reads                         |
| Datenvolumen     | Klein (max. ~1 Mrd. mögliche IBANs, realistisch: Tausende bis Millionen) |
| Konsistenz       | Nicht kritisch (Validierungsergebnis ist deterministisch)                |
| Relationen       | Keine (Stand jetzt — keine JOINs, keine Fremdschlüssel)                  |

### Optionen im Vergleich

#### Option A: PostgreSQL beibehalten ✅ (Empfehlung)

**Pro:**

- Bereits eingerichtet (Docker Compose, Flyway, Spring Data JPA, Tests)
- PK-Lookup auf `VARCHAR(34)` ist extrem schnell (B-Tree-Index, ~1 Page Read)
- Bietet Erweiterbarkeit: falls später Relationen hinzukommen (z.B. User → IBAN, IBAN → Validation-History), ist eine relationale DB der richtige Ort
- Spring Data JPA funktioniert nahtlos mit String-PKs
- Flyway-Migrationen bleiben identisch
- Für dieses Datenvolumen macht die DB-Technologie keinen messbaren Performance-Unterschied

**Contra:**

- Minimal „overpowered" für reines Key-Value — aber das ist kein realer Nachteil

#### Option B: Redis (Key-Value-Store)

**Pro:**

- Perfekt für Key-Value-Lookups (O(1) statt O(log n))
- In-Memory = extrem schnell (Sub-Millisekunde)
- Eingebauter TTL-Mechanismus für Cache-Expiry
- Einfaches Datenmodell (`SET iban:DE89... "{json}"`)

**Contra:**

- **Infrastruktur-Overhead**: Neuer Container in Docker Compose, neue Dependency (`spring-boot-starter-data-redis`), neues Konfigurationskonzept
- **Kein SQL**: Flyway-Migrationen entfallen → Schema-Versionierung verliert man
- **In-Memory-Kosten**: Alle Daten müssen in den RAM passen (bei Millionen IBANs unkritisch, aber konzeptionell anders als Disk-basiert)
- **Persistenz-Komplexität**: Redis ist primär ein Cache. Persistenz (RDB/AOF) ist möglich, aber weniger robust als PostgreSQL
- **Lernziel-Verlust**: JPA, Flyway, SQL-Migrations, Repository-Pattern — alles zentrale Lernziele — würden entfallen
- Nicht sinnvoll als alleinige Datenbank; eher als **Caching-Layer vor PostgreSQL**

#### Option C: Redis als Cache + PostgreSQL als Persistence

**Pro:**

- Best of both worlds: Redis für schnelle Cache-Lookups, PostgreSQL für dauerhafte Speicherung
- Spring Boot hat native Unterstützung (`@Cacheable` + `spring-boot-starter-cache` + Redis)

**Contra:**

- Deutlich erhöhte Komplexität (Cache-Invalidierung, zwei Datenquellen, Consistency)
- Für ein Demo-/Lernprojekt mit Tausenden von Einträgen massiv overengineered
- Lösung sucht ein Problem — die PostgreSQL-PK-Lookup-Latenz ist bereits <1ms

#### Option D: SQLite / H2 (Embedded DB)

Bereits in [decisions.md → ADR #4](decisions.md) bewertet und begründet abgelehnt. Die Argumente ändern sich durch den Umbau nicht.

### Fazit Datenbankwahl

**PostgreSQL beibehalten.** Das Zugriffsmuster wird durch den Umbau zwar cache-ähnlicher (Key-Value-Lookup), aber:

1. PostgreSQL kann Key-Value-Lookups auf einem B-Tree-Index in <1ms — für dieses Projekt ist das „schnell genug" um Größenordnungen.
2. Ein Technologiewechsel würde massive Infrastruktur- und Code-Änderungen verursachen, ohne messbaren Vorteil.
3. Die Lernziele (JPA, Flyway, Repository-Pattern, Docker-Compose-Orchestrierung) bleiben erhalten.
4. Falls später Relationen oder komplexe Queries hinzukommen (User-Accounts, Validierungs-Historie, Statistiken), ist PostgreSQL bereits da.

> **Faustregel**: Redis als Cache lohnt sich erst, wenn die Latenz von PostgreSQL (wenige ms) zum Bottleneck wird — bei Tausenden Requests/Sekunde. Für ein Demo-Projekt ist das irrelevant.

## 4. Auswirkungen und Änderungen — Schicht für Schicht

### 4.1 Flyway-Migration (`V1__initial_schema.sql` anpassen)

Da das Projekt noch nirgendwo produktiv läuft und keine bestehenden Daten existieren, wird keine separate V2-Migration benötigt. Stattdessen wird die initiale Migration `V1__initial_schema.sql` direkt auf das neue Schema angepasst:

```sql
-- V1__initial_schema.sql

CREATE TABLE ibans (
    iban              VARCHAR(34) PRIMARY KEY,
    bank_name         VARCHAR(255),
    bank_identifier   VARCHAR(20),
    valid             BOOLEAN NOT NULL,
    reason            VARCHAR(255),
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

> **Hinweis:** In Produktion würde man eine neue Migration (`V2__...`) erstellen, um das bestehende Schema zu transformieren. Da es hier noch keine Daten gibt, ist die direkte Änderung der V1-Migration der sauberste Weg — kein Migrationspfad nötig, kein Altdaten-Handling.

### 4.2 JPA Entity (`Iban.java`)

```java
@Entity
@Table(name = "ibans")
public class Iban {

    @Id  // IBAN ist jetzt der Primary Key — kein @GeneratedValue mehr
    @Column(nullable = false, length = 34)
    private String iban;

    @Column(length = 255)
    private String bankName;

    @Column(length = 20)
    private String bankIdentifier;

    @Column(nullable = false)
    private boolean valid;

    @Column(length = 255)
    private String reason;  // z.B. "IBAN zu kurz", null bei gültiger IBAN

    @Column(nullable = false)
    private Instant createdAt;

    @Column(nullable = false)
    private Instant updatedAt;

    protected Iban() {}

    public Iban(String iban, String bankName, String bankIdentifier,
                boolean valid, String reason) {
        this.iban = iban;
        this.bankName = bankName;
        this.bankIdentifier = bankIdentifier;
        this.valid = valid;
        this.reason = reason;
        this.createdAt = Instant.now();
        this.updatedAt = Instant.now();
    }

    // Getter + getReason() + getUpdatedAt()
}
```

**Wesentliche Änderungen:**

- `@Id` wandert von `Long id` auf `String iban`
- `@GeneratedValue` entfällt komplett (die IBAN wird vom Client geliefert, nicht von der DB generiert)
- `validationMethod` entfällt komplett
- Neues Feld `reason` für den Validierungsgrund bei ungültigen IBANs (nullable)
- Neues Feld `updatedAt` für Cache-Frische

> **TS-Analogie:** Statt `{ id: number, iban: string }` wird es `{ iban: string }` — die IBAN selbst ist der Identifier, wie ein `Map<string, IbanData>` statt `Array<{ id, iban, ...data }>`.

### 4.3 Repository (`IbanRepository.java`)

```java
// Vorher: JpaRepository<Iban, Long>   — Lookup by auto-ID
// Nachher: JpaRepository<Iban, String> — Lookup by IBAN-String
public interface IbanRepository extends JpaRepository<Iban, String> {
    // findById(iban) kommt gratis von JpaRepository
    // existsById(iban) kommt gratis von JpaRepository
}
```

Der generische Typ-Parameter ändert sich von `Long` auf `String`. Alle Standard-Methoden (`findById`, `existsById`, `save`, `findAll`, `deleteById`) funktionieren automatisch mit dem neuen String-PK.

> **TS-Analogie:** Wie wenn ein Prisma-Model von `id Int @id @default(autoincrement())` auf `iban String @id` wechselt.

### 4.4 Controller (`IbanController.java`)

Der Controller bekommt die **Lookup-Logik** — die wichtigste fachliche Änderung:

```java
@PostMapping
public ResponseEntity<IbanResponse> validateAndSaveIban(
        @Valid @RequestBody IbanRequest request) {

    String normalizedIban = normalize(request.iban());

    // ── NEU: Lookup — bereits bekannte IBAN direkt zurückgeben ──
    Optional<Iban> existing = ibanRepository.findById(normalizedIban);
    if (existing.isPresent()) {
        Iban cached = existing.get();
        return ResponseEntity.ok(new IbanResponse(
            cached.isValid(),
            cached.getIban(),
            cached.getBankName(),
            cached.getBankIdentifier(),
            cached.getReason()
        ));
    }

    // ── Cache-Miss: Validierung durchführen ──
    IbanResponse response = validateIban(normalizedIban);

    ibanRepository.save(new Iban(
        response.iban(),
        response.bankName(),
        response.bankIdentifier(),
        response.valid(),
        response.reason()
    ));

    return ResponseEntity.ok(response);
}
```

**Design-Entscheidung: Wo lebt die Lookup-Logik?**

| Option     | Beschreibung                                           | Bewertung                             |
| ---------- | ------------------------------------------------------ | ------------------------------------- |
| Controller | Lookup im Controller, Validate im Service              | Einfach, passend für dieses Projekt   |
| Service    | Neuer `IbanService` der Lookup + Validate orchestriert | Besser bei komplexerer Business-Logik |
| Repository | Custom Query im Repository                             | Falsche Schicht für Orchestrierung    |

Für dieses Projekt ist der Controller der richtige Ort: der Lookup ist kein Business-Logic-Detail, sondern eine **Infrastruktur-Optimierung** (Cache-Check vor Berechnung). Alternativ könnte man einen dedizierten `IbanLookupService` einführen, falls die Logik komplexer wird (z.B. TTL-Prüfung, Revalidierung).

### 4.5 DTOs / Response Records

```java
// IbanListEntry: id und validationMethod entfallen, reason kommt hinzu
// Vorher:
public record IbanListEntry(Long id, String iban, String bankName, ...)

// Nachher:
public record IbanListEntry(String iban, String bankName,
    String bankIdentifier, boolean valid, String reason)
```

Das `id`-Feld und `validationMethod` werden aus allen DTOs entfernt. Dafür kommt `reason` hinzu, damit der Fehlergrund bei ungültigen IBANs auch aus dem Cache zurückgegeben werden kann.

## 5. Offene Design-Fragen

### 5.1 Cache-Invalidierung / TTL

IBANs sind statisch (eine IBAN wird nicht plötzlich ungültig). Aber der `bankName` könnte sich ändern (Banken fusionieren, benennen sich um). Optionen:

| Strategie                   | Umsetzung                                                                     | Sinnvoll?                                 |
| --------------------------- | ----------------------------------------------------------------------------- | ----------------------------------------- |
| Kein TTL                    | Einmal gespeichert = für immer                                                | Für dieses Lernprojekt völlig ausreichend |
| TTL per Cronjob             | Einträge älter als X Tage löschen                                             | Overengineered                            |
| Manuelles Revalidieren      | DELETE-Endpoint pro IBAN + erneutes POST                                      | Könnte als Feature interessant sein       |
| `updated_at` + Soft-Refresh | Bei Lookup prüfen ob `updated_at` > 30 Tage, dann im Hintergrund revalidieren | Zu komplex für Lernprojekt                |

**Empfehlung:** Kein TTL. Das `updated_at`-Feld vorbereiten, aber nicht aktiv nutzen. Falls gewünscht, einen `DELETE /api/ibans/{iban}`-Endpoint ergänzen, der einen Cache-Eintrag manuell löscht.

### 5.2 Concurrency — Zwei gleichzeitige Anfragen für dieselbe IBAN

**Szenario:** Zwei Requests für `DE89...` kommen gleichzeitig. Beide machen einen Lookup, beide finden nichts, beide validieren, beide versuchen zu INSERT-en.

**Lösung:** PostgreSQL's `ON CONFLICT` (Upsert):

```java
// In JPA: save() auf eine Entity mit existierendem PK macht automatisch UPDATE statt INSERT.
// Spring Data JPA prüft intern, ob die Entity "neu" ist (isNew()).
// Bei einem String-PK muss man `Persistable<String>` implementieren oder
// `@Version` verwenden, damit JPA weiß, ob merge oder persist.
```

Pragmatische Lösung für dieses Projekt: Da das Ergebnis für dieselbe IBAN deterministisch ist, ist ein doppelter INSERT harmlos — der zweite überschreibt den ersten mit identischen Daten. Die `Iban`-Entity sollte `Persistable<String>` implementieren.

## 6. Natürlicher vs. Surrogate Key — Abwägung

Die Entscheidung, die IBAN als Primary Key zu verwenden, ist ein bewusster Trade-off:

### Pro natürlicher Key (IBAN als PK)

- **Fachlich korrekt**: Eine IBAN ist weltweit eindeutig — sie _ist_ der natürliche Identifier
- **Kein Join nötig**: Kein Lookup über einen bedeutungslosen Integer, um an die IBAN zu kommen
- **Einfachere Queries**: `WHERE iban = 'DE89...'` statt `WHERE id = 42` und dann `iban` rauslesen
- **Weniger Spalten**: Die `id`-Spalte entfällt → weniger Storage, weniger Index-Overhead
- **Bessere Semantik**: Die Tabelle repräsentiert tatsächlich das, was der Name sagt

### Contra natürlicher Key

- **String-PK ist größer als BIGINT**: 34 Bytes vs. 8 Bytes. Bei Millionen Zeilen spürbar — bei Tausenden irrelevant
- **Nicht änderbar**: Falls die IBAN sich ändert (unrealistisch), müsste der PK geändert werden. Bei Fremdschlüsseln kaskadierend
- **JPA-Eigenheit**: Spring Data JPA unterscheidet zwischen `persist` (neue Entity) und `merge` (vorhandene Entity) anhand des PK. Bei auto-generierten IDs ist `id == null` → neu. Bei String-PKs muss man `Persistable<String>` implementieren, damit JPA weiß, ob INSERT oder UPDATE

### Bewertung

Für diesen Use Case ist der natürliche Key klar die bessere Wahl. Die Nachteile sind entweder irrelevant (Datenvolumen) oder mit minimalem Aufwand lösbar (`Persistable`). Der Surrogate Key (`id BIGINT`) hat hier keinen Mehrwert — er existierte nur, weil das bisherige Modell ein Validierungs-Log war, keine Entitäts-Tabelle.

## 7. Zusammenfassung — Checkliste

### Backend

- [ ] **Flyway-Migration** `V1__initial_schema.sql` direkt anpassen
  - `iban` als PK, kein `id`, kein `idx_ibans_iban`-Index
  - `reason`-Spalte hinzufügen (`VARCHAR(255)`, nullable)
  - `updated_at`-Spalte hinzufügen
  - `validation_method`-Spalte entfernen
- [ ] **Entity** `Iban.java` anpassen
  - `@Id` auf `String iban`, `@GeneratedValue` entfernen
  - `Persistable<String>` implementieren (für korrektes INSERT/UPDATE-Verhalten)
  - `validationMethod`-Feld entfernen
  - `reason`-Feld hinzufügen
  - `updatedAt`-Feld + Getter hinzufügen
- [ ] **Repository** `IbanRepository.java` anpassen
  - Generischen Typ von `Long` auf `String` ändern
- [ ] **Controller** `IbanController.java` anpassen
  - Lookup-Logik vor Validierung einfügen (`findById`)
  - `IbanListEntry`-Record: `id`-Feld entfernen
  - `normalize()`-Methode im Controller (oder als Utility) bereitstellen
  - Optional: `DELETE /api/ibans/{iban}`-Endpoint hinzufügen
- [ ] **Tests** anpassen
  - Neue Tests: Cache-Hit, Cache-Miss
  - Bestehende Tests: `id`-Feld aus Mocks entfernen
  - `IbanRepository` Mock: `findById(String)` statt `findById(Long)`

### Frontend

- [ ] **`api.ts`**: `IbanListEntry.id` entfernen
- [ ] **`IbanInput.tsx`**: `key={entry.id}` → `key={entry.iban}`
- [ ] **`IbanInput.test.tsx`**: Mock-Daten anpassen (kein `id`)

### Dokumentation

- [ ] **`decisions.md`**: Neuen ADR hinzufügen ("IBAN als natürlicher Primary Key")
- [ ] **`README.md`**: API-Dokumentation aktualisieren (Lookup-Verhalten beschreiben)

### Nicht ändern

- **PostgreSQL** bleibt (keine DB-Migration nötig)
- **Docker Compose** bleibt unverändert
- **Flyway** bleibt (V1-Migration wird direkt angepasst, da keine Bestandsdaten)
- **Spring Data JPA** bleibt (funktioniert mit String-PKs)
- **Controller → Service → Repository** Schichtung bleibt
- **IbanValidationService** und **ExternalIbanApiService** bleiben unverändert
