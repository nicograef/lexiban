IBAN Validierung für Selbstständige und kleine Unternehmen

Nico Gräf // 2026

Aufgabe

Eine React SPA, die über eine REST-API
mit einem Java Spring Boot Backend kommuniziert,
um IBANs zu validieren.

IBAN Fachlichkeit

  • Internationale Kontonummer (max. 34 Zeichen)
  • Definiert im ISO 13616 Standard
  • Länderkennung nach ISO 3166-1
  • Prüfziffern zur Validierung (Modulo 97 nach ISO 7064)
  • BBAN individuell je Land

IBAN Validierung

  • Bereinigen und Normalisieren
  • Länge & Struktur prüfen
  • Modulo 97 Algorithmus für Prüfziffern
  • Banknamen-Auflösung über BLZ oder externe API

Umsetzung

  • Freie IBAN-Eingabe — Leerzeichen erlaubt, automatische 4er-Gruppierung nach DIN 5008
  • Eigene Modulo-97-Validierung — keine Library, eigene Implementierung nach ISO 13616
  • Banknamen-Auflösung — drei vordefinierte Banken + Fallback auf externe API
  • Persistenz — validierte IBANs in PostgreSQL mit Cache-Lookup
  • IBAN-Liste — gespeicherte IBANs im Frontend anzeigen

DevOps

  • Lokale Entwicklung — Makefile & Docker: PostgreSQL + Spring Boot + Vite Dev-Server
  • VPS — Docker Compose mit Certbot für HTTPS
  • AWS — Cloudfront + S3, RDS + Proxy, Lambda + API Gateway
  • CI/CD — Github Actions: Build, Test, Validate, Deploy

Backend

  • Flyway Schema Migration — Hibernate ddl-auto=validate
  • Java Records für DTOs — Request & Response, Immutable, kein Lombok, plain Java
  • Maven statt Gradle — Einfacher, kein custom Build notwendig
  • Constructor Injection — Explizite Abhängigkeiten, testbar ohne Spring-Kontext
  • IBAN als natürlicher PK — Jede IBAN existiert genau einmal, Cache-Hit bei Wiederholung

Design-Prinzipien

  • Dünner Controller — kein Business-Logik, nur HTTP ↔ Service Mapping, Validierung via @Valid
  • Value Object IbanNumber — self-normalizing Record, entfernt Sonderzeichen, Uppercase, strukturelle Validierung
  • Strategy Pattern — IbanValidator-Interface mit LocalIbanValidator und OpenIbanValidator, Fallback via Optional<ValidationResult>
  • Graceful Degradation — externer API-Call in try/catch, App funktioniert immer
  • Zentrales Error Handling — @RestControllerAdvice mit konsistenten Error-Responses (400/500)
