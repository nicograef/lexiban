Lexiban

IBAN Validierung für Selbstständige und kleine Unternehmen

Nico Gräf // 2026

Aufgabe

Eine React SPA, die über eine REST-API mit einem Java Spring Boot Backend kommuniziert, um IBANs zu validieren.

IBAN Fachlichkeit

  • Internationale Kontonummer (max. 34 Zeichen)
  • Definiert im ISO 13616 Standard
  • Länderkennung nach ISO 3166-1
  • Prüfziffern zur Validierung (Modulo 97 nach ISO 7064)
  • BBAN individuell je Land

  ├─────┤    ├─────┤  ├────────────────┤ ├─────────────────────┤
Ländercode  Prüfziffern   Bankleitzahl         Kontonummer

IBAN Validierung

  • Bereinigen und Normalisieren
  • Strukturprüfung: Länge, erlaubte Zeichen, Länderspezifika
  • Modulo 97 Algorithmus für Prüfziffern
  • Banknamen-Auflösung über BLZ oder externe API

Umgesetzte Anforderungen

  1. Freie IBAN-Eingabe — Leerzeichen erlaubt, automatische 4er-Gruppierung
  2. Eigene Modulo-97-Validierung — keine Library, eigene Implementierung nach ISO 13616
  3. Banknamen-Auflösung — drei vordefinierte Banken + Fallback auf externe API
  4. Persistenz — validierte IBANs in PostgreSQL mit Cache-Lookup
  5. IBAN-Liste — gespeicherte IBANs im Frontend anzeigen

Happy Path

  1. IBAN eingeben → automatische 4er-Formatierung
  2. „Prüfen" → Ergebnis: gültig, Bankname und BLZ
  3. IBAN erscheint in der gespeicherten Liste
  4. Gleiche IBAN erneut → Cache-Hit, sofortige Antwort

Edge Cases

  • Ungültige IBAN → Fehlermeldung
  • Leere Eingabe → HTTP 400
  • Unbekannte Bank → Fallback auf openiban.com
  • Wiederholte Anfrage → kein erneuter Validierungslauf

Design-Prinzipien

Dünner Controller

Kein Business-Logik, nur HTTP ↔ Service Mapping.
Validierung via @Valid.


Value Object IbanNumber

Self-normalizing Record: entfernt Sonderzeichen, Uppercase, strukturelle Validierung.
Wer ein IbanNumber hat, weiß dass es normalisiert ist.

Design-Prinzipien

Strategy Pattern für Validierung

Ein IbanValidator-Interface, zwei Implementierungen:

  • LocalIbanValidator — Länderlänge + Mod-97 + BLZ-Lookup
  • OpenIbanValidator — REST-Call an openiban.com

Beide geben Optional<ValidationResult> zurück — empty() heißt „nächster Validator ist dran".

Design-Prinzipien

Graceful Degradation

Externer API-Call komplett in try/catch.
Die App funktioniert immer, schlimmstenfalls ohne Banknamen.


Zentrales Error Handling

@RestControllerAdvice mit konsistenten Error-Responses
für Validierungsfehler (400) und unerwartete Fehler (500).
